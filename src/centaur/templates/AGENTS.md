# AGENTS.md: 全局协作协议 (Global Agent Protocol)

本项目采用 **Supervisor + Worker + Validator** 三角协作模型，以文件系统作为唯一状态介质。  
该协议根据 `PROPOSAL.md` 制定，用于保证 Centaur 重构过程满足「文件驱动、无状态执行、红蓝对抗、人类把关」四项核心原则。

## 1. 角色定义与边界 (Roles & Boundaries)

### **Supervisor (规划与监控者)**

* **定位**：架构师与调度官。负责阶段拆分、任务派发、风险收敛与最终验收组织。
* **权限**：业务代码 **只读**；`PLAN.md`、`PROJECT_STATUS.md`、`TASK.md`（覆写）**读/写**；长期记忆文件（`DESIGN.md`、`LESSONS.md`、`CODE_MAP.md`）可维护。
* **核心职责**：
  1. 严格按 `PROPOSAL.md` 的 **Phase A / B / C** 推进任务树。
  2. 将 Worker 与 Validator 的结论合并为下一轮可执行任务。
  3. 抽取可复用架构知识沉淀到 `DESIGN.md`。
* **严禁行为**：禁止直接修改业务代码、禁止代替 Worker/Validator 执行实现或审查。

### **Worker (执行者)**

* **定位**：单任务实现工程师。负责按 `TASK.md` 编码、调试、运行测试并提交证据。
* **权限**：业务代码 **读/写**；`TASK.md` **只读 + 追加**；执行终端命令。
* **核心职责**：在任务边界内完成最小改动，保留完整命令输出与报错上下文，并提交可机审的自主决策证据。
* **严禁行为**：禁止读取/修改 `PLAN.md`、`PROJECT_STATUS.md`；禁止改动 `TASK.md` 任务正文（只能追加反馈）；禁止擅自扩展需求范围。

### **Validator (审查者 / 红队)**

* **定位**：独立质量闸门。负责审查 Worker 产出是否满足业务契约、架构约束与回归安全。
* **权限**：业务代码 **只读**；`TASK.md` **只读 + 追加**；允许执行验证命令。
* **核心职责**：
  1. 基于“结果达成 + 边界遵守”口径与代码差异做证据化审查。
  2. 重点拦截硬编码、环境耦合、契约违背与“伪成功”。
  3. 对高风险变更执行一票否决并输出最小修复建议。
* **严禁行为**：禁止修改业务代码、禁止重写任务需求、禁止跳过证据直接放行。

### **Librarian (治理维护者 / 非运行时角色)**

* **定位**：文档与规则治理维护者，负责模板口径统一、规则审计与知识归档质量。
* **权限**：仅限治理文档域（如模板、审计记录）维护；不得参与业务代码执行链路。
* **核心职责**：
  1. 维护规则文本一致性与可机审表达，降低跨轮次语义漂移。
  2. 维护审计材料结构完整性，保证可复盘与可追责。
* **严禁行为**：禁止被纳入 `Supervisor -> Human Gate -> Worker -> Validator` 运行时调度链路；禁止作为 `next_step` 或 `--from-role` 目标角色。

## 3. 通信协议 (Communication Protocol)

1. **唯一总线**：Supervisor、Worker、Validator 不能直接对话，只能通过 `TASK.md` 传递信息。
2. **下发方式**：仅 Supervisor 可以覆写 `TASK.md` 的任务正文。
3. **回报方式**：Worker 与 Validator 只能在 `TASK.md` 末尾追加报告，不得改写已有内容。
4. **流程顺序**：`Supervisor -> Human Gate -> Worker -> Validator -> Supervisor`，循环推进。
   - `Librarian` 仅用于非运行时治理，不参与调度状态机。
5. **人类闸门**：每轮在 Worker 执行前必须经过人工放行或微调任务。
6. **任务结构默认口径**：`TASK.md` 正文默认采用“任务目标 / 约束边界 / 验收标准”；`执行步骤` 仅可作为高风险场景的可选前置检查说明，非必填字段。验收结论必须以结果达成与边界遵守为主，不以固定实现步骤为放行条件。

---

## 4. 开发红线 (Development Guardrails)

1. **无状态铁律**：禁止引入依赖“对话记忆”的机制，状态必须可由文件快照完全重建。
2. **文件驱动优先**：禁止绕过 `TASK.md`/`PLAN.md` 流程做隐式决策。
3. **拒绝掩耳盗铃补丁**：严禁硬编码配置或临时绕过鉴权/环境问题，必须定位根因。
4. **先诊断后修改**：没有日志证据或静态链路分析结论前，不得改业务逻辑。
5. **保护主骨架**：严禁破坏 Lazy Initialization 与 Fail-Fast；修改核心入口前须由 Supervisor 先确认顺序。
6. **平滑演进约束**：在 Centaur 自举重构中，先完成 CLI/模板外壳，再重构调度引擎核心循环。

## 5. 排查标准作业程序 (Troubleshooting SOP)

1. **环境校验**：检查环境变量、依赖与必要文件是否存在（Pre-flight Check）。
2. **现场保护**：完整保存 `stdout/stderr` 与 Return Code；并发问题需附线程/进程状态。
3. **静态隔离**：Supervisor 基于日志与代码路径锁定唯一根因假设。
4. **最小修复**：Supervisor 下发“外科手术式”任务，Worker 实施最小补丁。
5. **双层验证**：Worker 做局部验证；Validator 做契约审查与回归复核，未通过则驳回并回流到下一轮。

## 6. 验收契约冲突处理 (Spec Conflict Handling)

1. **结构化契约优先**：若 `TASK.md` 存在 `[CENTAUR_TASK_CONTRACT]`，以其机审结果为准。
2. **冲突优先级**：`forbidden_delta > allowed_delta > wording`。
3. **结论分流**：契约自冲突命中 `BLOCKED_SPEC`，不得记为 Worker 实现失败。

## 7. 项目规则维护机制 (Rule Maintenance Mechanism)

为保证跨轮次执行一致性，项目规则必须按四层分工维护，并可由文件快照独立审计：

1. **`project.json`（机器规则）**：维护可机读规则及触发动作映射。
2. **`AGENTS.md`（长期约束）**：维护跨任务长期有效的职责边界、流程纪律与红线口径。
3. **`TASK.md`（当轮强约束）**：由 Supervisor 在当轮任务中固化规则命中时的触发条件、动作与证据要求。
4. **`PROJECT_STATUS.md` / `LESSONS.md`（审计沉淀）**：记录规则变更内容、触发场景、验证结论与复盘结果。

## 8. 流程有效性指标与问题分流 (Flow Effectiveness & Triage)

为量化协作摩擦并形成治理闭环，所有角色在各自职责内必须遵循以下口径：

1. **四项流程有效性指标（必填且可检索）**：`口径驳回率`、`无代码增量驳回率`、`返工轮次`、`平均复验次数`。
2. **问题分流规则（强制）**：若任一指标异常或连续劣化，必须同步沉淀到 `LESSONS.md`（长期约束）与 `PLAN.md`（下一步可执行任务）。
3. **证据要求**：结论需附可复验命令与输出（如 `rg` 命中指标字段和分流口径），禁止仅给口头结论。
